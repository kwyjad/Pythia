# Pythia Codebase Audit — July 2026

A full-codebase audit covering (A) functional issues, (B) bugs, and (C)
structural improvements, with emphasis on the recent scoring/bucket work
(PRs #774–#777) plus deep passes over the forecaster and Resolver systems.
Three parallel review sweeps were run (forecaster chain; Resolver +
post-forecast pipeline; API + Horizon Scanner); the highest-severity
candidates were then hand-verified against source before fixing. This
document records what was found, what was fixed, what was deliberately
deferred, and which reported findings turned out to be false positives —
so future sessions don't re-litigate them.

Overall verdict on the recent work: **the month-anchoring, bucket-derivation,
and score-family-separation changes held up well** — validation in those
paths is strong and none of the serious findings were in the PR #774–#777
code. The serious findings were elsewhere (API input handling and the
Resolver precedence path).

## Fixed in this audit (branch `audit/2026-07-fixes`)

### F1 — SQL injection via `iso3` and `forecaster_run_id` query params (HIGH, security)
`pythia/api/app.py`. The `iso3` param on `/v1/resolver/source_inventory` and
`/v1/resolver/source_data` was interpolated into SQL as
`f"iso3 = '{iso3.upper()}'"` with no validation; the `forecaster_run_id`
param (8+ endpoints) was interpolated into the `fc_run_filter` CTE the same
way. Fixed: `_validate_iso3_param()` (regex `^[A-Za-z]{3}$`, 400 otherwise)
plus bound `?` parameters for iso3; strict pattern validation for
forecaster_run_id at the `_run_filter_cte` chokepoint. Tests:
`pythia/tests/test_api_input_validation.py`.

### F2 — Precedence policy silently non-functional in the pipeline path (HIGH, data integrity)
`resolver/tools/precedence_config.yml` + `precedence_engine.py`. Three
compounding problems: (1) the tier-0 entry `ifrc_montandon` never matched
the publisher string the IFRC connector actually writes (`IFRC`), so IFRC —
the documented tier-0 source for natural-hazard PA — fell to the default
(worst) tier; (2) the engine's `resolve_facts_frame` reads config keys
`tiebreak`/`defaults`/`metrics` which did not exist in the YAML, so
within-tier winners were chosen alphabetically by `(source, run_id)` — no
recency preference, and a NULL-value row could beat a non-null one; (3) the
YAML's `metric_preference`/`source_precedence`/`conflict_rule` keys are read
only by the legacy CLI path, not the pipeline path. Fixed: `ifrc` added to
tier 0, `tiebreak: [as_of_desc, value_nonnull, source_alpha]` and
`defaults: {require_nonnegative, prefer_full_row_coverage}` added to the
YAML (mirroring the long-tested fixture config), and the engine now defaults
to `[as_of_desc, value_nonnull]` when the config omits the key. Tests:
`resolver/tests/test_precedence_pipeline_config.py` exercises the REAL
production config. **Note: this intentionally changes which rows win
`facts_resolved` on the next pipeline run.**

### F3 — FATALITIES zero-defaults for countries outside ACLED's universe (MEDIUM-HIGH)
`pythia/tools/compute_resolutions.py`. Zero-defaulting was gated only on
global month coverage, so a country ACLED has never reported still resolved
to FATALITIES=0. New `_countries_with_source_data()` universe gate: a
country must appear at least once (any month) in the metric's source tables
to zero-default. Applied to FATALITIES only — EVENT_OCCURRENCE is exempt
because GDACS coverage is satellite-global and only writes rows where events
occurred (gating it would permanently un-resolve low-disaster countries).
New counter `skipped_outside_source_universe` in the run summary.

### F4 — Centroid version stamped from the wrong hazard (MEDIUM)
`pythia/tools/compute_scores.py` `_get_centroid_version()` ignored its
`hazard_code` parameter (metric-only MAX across all hazards). Now mirrors
`_load_centroids`' hazard-specific → wildcard lookup order.

### F5 — Binary pipeline accepted/wrote partial month coverage (MEDIUM)
`forecaster/binary_prompts.py` + `forecaster/cli.py`. A model response
missing expected months passed through partially and could leave silent
month gaps in `forecasts_ensemble`. Now: `parse_binary_response` rejects an
incomplete response outright (model-level failure), and the writer path has
a belt-and-braces `insufficient_month_coverage` gate mirroring the SPD
convention (never write a partial window).

### F6 — Calibration weights cache: unlocked, never reset (MEDIUM)
`forecaster/cli.py` `_CALIB_WEIGHTS_CACHE` is read/written from SPD worker
threads; the unguarded check-then-set raced duplicate DB reads, and a
long-lived process never picked up recomputed weights. Now guarded by
`_CALIB_WEIGHTS_LOCK` and cleared via `_reset_calibration_weights_cache()`
at v2 pipeline start.

### F7 — `include_test` filter inconsistency in track counts (MEDIUM)
`pythia/api/app.py` `/v1/performance/scores`: the track-split query omitted
the test filter that the total-count query applied, so track1+track2 could
disagree with the total when test data exists.

### F8 — Silent exception swallowing (observability)
Warnings added (behavior unchanged, still non-fatal): GDACS current-alerts
load and reasoning-trace serialization in `forecaster/cli.py`;
`resolution_rates` query failure and `/v1/version` staleness probes in
`app.py`; ACLED CAST fetch failure upgraded to `LOG.error`; `_rps` warns on
<2-bucket input; `enrich.py` event_id fallback warns on empty components.

### F9 — `datetime.utcnow()` deprecation (LOW)
Replaced with `datetime.now(timezone.utc).replace(tzinfo=None)` in the
recurring pipeline modules (forecaster cli/llm_logging, horizon_scanner
core/db_writer/llm_logging, pythia tools/db/gdelt, resolver
duckdb_io/acled_client). **Remaining long tail** (peripheral, fix
opportunistically): `horizon_scanner/seasonal_tc/*` scrapers,
`pythia/web_research/backends/*` (fetched_at stamps),
`scripts/dump_pythia_debug_bundle.py`, `scripts/ci/build_llm_context.py`,
`resolver/snapshot/builder.py`, `resolver/ingestion/_stub_utils.py`,
`resolver/ingestion/run_all_stubs.py`, `resolver/tools/freeze_snapshot.py`,
`resolver/tools/build_forecaster_features.py`,
`resolver/tools/generate_schemas_md.py`, `resolver/cli/simple_snapshot.py`,
`pythia/web_research/types.py`, `pythia/tools/score_views.py` (done),
`docs/prompts/**` (snapshots — never edit).

### F10 — Silent PA bucket-count fallback (LOW)
The 8 `n_buckets_for(metric) or len(SPD_CLASS_BINS)` sites in
`forecaster/cli.py` now route through `_n_buckets_for_metric()`, which logs
a warning whenever the PA fallback fires for an empty/unknown metric.

### S1–S3 — Structural cleanups
- `pythia/tools/_db_utils.py`: single home for the rollback-safe DuckDB
  helpers previously duplicated (with drift — the resolutions/scores copies
  lacked the aborted-transaction ROLLBACK scrub) across the four
  post-forecast tools.
- Dead centroid constants removed from `forecaster/ensemble.py` (all three)
  and `forecaster/aggregate.py` (FATALITIES variant).
- `NUM_HORIZONS = 6` single-sourced in `pythia/buckets.py`, imported by
  `compute_resolutions.py` and `forecaster/cli.py` window-bound checks.

## Investigated — NOT bugs (do not re-fix)

- **`order` param "injection" in `/v1/resolver/source_data`**: the `order`
  string comes from the hardcoded `_SOURCE_REGISTRY` spec, not user input.
  The column-existence validation is defense-in-depth, not a boundary.
- **RC threshold sync** between `_select_tail_pack_hazards`
  (horizon_scanner.py) and `adversarial_check.py`: verified in sync — both
  gate on `level < 1` via the shared `compute_level()`, so drift is
  structurally prevented.
- **EMA centroid DELETE+INSERT "race"** in `compute_bucket_centroids.py`:
  calibration runs single-threaded in CI; the DELETE breadth was already
  fixed (see CLAUDE.md known failure modes).
- **Calendar cutoff "off-by-one"** in `_calendar_cutoff`: works as designed
  in combination with the per-metric data-freshness cutoff; absolute
  correctness at month boundaries is bounded by source publication lag.
- **`_month_index_for_label` hardcoding 6**: the 6-month window is a system
  invariant, not a per-question property. Addressed by single-sourcing
  `NUM_HORIZONS` (S3), not by parameterizing the window per question.
- **`test_event_occurrence_no_data_defaults_to_zero`** semantics: a country
  with zero GDACS rows genuinely means "no qualifying events" — this is why
  EVENT_OCCURRENCE is exempt from the F3 universe gate.

# Follow-up audit — 2026-07-07 (post-merge sweep of PRs #780–#783)

A second sweep covering the four large merges that landed after the audit
above: Sibyl (#782), the API memory/OOM fixes (#781), the model-registry
refactor (#780), and the model lineup/per-1M cost update (#783) — plus a
fresh cross-cutting pass. Three parallel review agents; every fix below was
hand-verified against source before changing anything. Fixes landed on
branch `claude/codebase-audit-dld6wd`.

**Overall verdict**: Sibyl is clean — pooling math, bucket discretization,
all four DB INSERTs, budget-cap boundaries, and cost accounting verified
correct; its findings were leakage/telemetry hardening. The serious bugs
were in the lineup update (dead JSON-repair fallback), the API connection
layer (a NEW stale-DB root cause), and two pipeline gates.

## Fixed in this follow-up

### FU1 — gpt-5.4 JSON-repair fallback dead: temperature sent, API rejects it (MEDIUM)
`forecaster/providers.py`. `_OPENAI_NO_CUSTOM_TEMPERATURE` listed only
`gpt-5-mini`/`gpt-5.4-mini`; when `reasoning_effort` is None (the
`hs_fallback` spec carries no thinking value) `call_openai` sent
`temperature` to `gpt-5.4` → HTTP 400 → every triage/RC JSON repair failed.
Now prefix-matched (`_OPENAI_NO_TEMPERATURE_PREFIXES = ("gpt-5",)`),
mirroring the Anthropic rule, with body-level regression tests.

### FU2 — DuckDB per-path instance caching defeated every connection reopen (HIGH)
`pythia/api/core.py` + `app.py` force_sync. DuckDB caches database
instances per path within a process: opening a "new" connection while the
old one is still open returns a handle to the SAME cached instance — which
still reads the pre-`os.replace()` inode. Both reopen sites
(`_maybe_refresh_db`, `/v1/admin/force_sync`) used open-new-then-close-old,
so the "reopened" connection never saw a swapped DB file (verified
empirically on DuckDB 1.5.4; the `_execute` retry path already used the
correct close-first order). This is the likely residual root cause of the
recurring dashboard-frozen-on-stale-DB incidents documented in CLAUDE.md.
New `core._swap_read_connection()` (close FIRST, then open) used by both.

### FU3 — /v1/version staleness probes pinned pre-refresh data (MEDIUM)
`pythia/api/app.py` `_staleness_probes` cached under the raw on-disk mtime
while computing through the throttled (possibly still-old) connection: the
first request after a publish cached pre-swap probes under the new DB
version until the NEXT publish. Now keyed on `core._READ_CON_MTIME` (the
mtime the connection was opened with). The download-export cache had the
same defect plus an `int()` mtime truncation (two swaps in the same second
collided) — same fix + full-precision mtime tag.

### FU4 — Brave breaker gate bypassed for priority hazards (MEDIUM)
`scripts/create_questions_from_triage.py` recomputed
`need = need_full_spd OR tier=="priority"`, so a priority hazard whose
`need_full_spd` the HS breaker gate had cleared (Brave budget exhausted,
zero grounding) was forecast anyway. The stored `need_full_spd` column is
now the single source of truth. Regression test added.

### FU5 — Manual dispatch of Compute Resolutions / Compute SPD Scores broken (MEDIUM)
Both workflows hardcoded `RUN_ID=${{ github.event.workflow_run.id }}`
(empty on dispatch, no discovery fallback) — contradicting the CLAUDE.md
claim that scores falls back to canonical discovery. compute_resolutions
also ran on FAILED upstream completions (no job-level `if:` guard). Fixed
via FU8 + the guard.

### FU6 — Sibyl backtest leak-filter gaps (MEDIUM, backtest-only) + telemetry (LOW)
`sibyl/leakage.py`: Brave's relative `page_age` forms ("2 weeks ago")
bypassed the post-asOf filter (now parsed, resolved against fetch time);
populated-but-unparseable dates are now dropped in backtest (new
`dropped_undatable` counter) — undated sources are still kept (freshness
range + snippet classifier remain their guards). `sibyl/tools.py`: fetch
leak scan now covers the full text returned to the model (was first 2000 of
6000 chars). `sibyl/run.py`: `budget_capped` is now also set when the cap
fires during the LAST question's trials. `pythia/api/routes/sibyl.py`:
`_latest_sibyl_run_id` table-guards `sibyl_runs` (never-500 contract).

### FU7 — Same-epoch question re-run left stale metadata (LOW)
`scripts/create_questions_from_triage.py` refreshed only `hs_run_id`/`track`
on existing rows; `pythia_metadata_json` (embedded hs_run_id/tier/
triage_score) now refreshes in the same UPDATE.

### FU8 — Canonical-DB download consolidated into a composite action (structural)
New `.github/actions/download-canonical-db` — single home of the candidate
list (HS Triage, Sibyl, Resolver Update, Ingest Structured Data, Compute
Resolutions/Scores/Calibration), with forced mode, Path A (triggering run →
discovery fallback), Path B discovery, per-candidate `validate-cmd`
(db_signature), retries, WAL cleanup. Nine workflows migrated (~890
duplicated lines removed); `run_horizon_scanner.yml` and
`publish_latest_data.yml` stay intentionally bespoke (override inputs /
regression-guard outputs) with sync-pointer comments. This also fixed FU5
and the peripheral candidate-list gaps (Sibyl missing from purge_hs_run,
resolver-initial-backfill, ingest-nmme). Dead OpenRouter env block removed
from manual_test.yaml.

### FU9 — Single-sourcing refactors (structural)
`prompts._forecast_month_keys_from_question` is now a wrapper over
`month_utils._anchor_month_for_question` + `_expected_months` (sibyl
already used them); `STANDARD_MODEL_PREFERENCE` moved to `sibyl/config.py`
(import-light) and imported by `sibyl/spd.py` + the API route.

## Investigated — NOT bugs (do not re-fix)

- **Calibration weights invalidated by the display-name change (#783)**:
  historical `calibration_weights.model_name` rows don't match the new
  model-id member keys, so the first post-deploy runs aggregate unweighted
  until calibration recomputes. This is the DOCUMENTED model-swap behavior
  (new names start at weight 1.0) — a transition window, not a bug.
- **Double publish per HS Triage completion**: publish triggers on HS
  Triage directly AND via run_sibyl's dispatch. Benign — the publish
  freshness/regression guard soft-skips the staler candidate. Removing the
  direct trigger would lose publishes when Sibyl fails; leave as is.
- **Sibyl divergence pairing across standard re-runs**: sibyl rows written
  under an older standard run_id remain scored there — internally
  consistent per-era pairing, not double-counting.
- **Export FileResponse vs sweeper TOCTOU**: 60s age grace narrows the
  window; residual risk accepted (Linux fd semantics survive unlink).
- **Current-mtime export cache files are never swept**: bounded by param
  cardinality on an ephemeral tmp dir; accepted.
- **`_CALIB_WEIGHTS_CACHE` identity across the cli.py decomposition**:
  verified held (cli imports the dict object; reset clears in place).
- **Month-anchoring consistency**: month_utils / prompts / sibyl all anchor
  window_start-first, target−5 fallback — verified consistent (and now
  single-sourced, FU9).
- **`AGGREGATE_MODEL_NAMES` includes `'sibyl'`** — verified; Sibyl is
  scored but excluded from the member-weight softmax as designed.
